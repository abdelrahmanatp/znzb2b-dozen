## Overview

The Google Sheets workbook is the operational database for the 521-prospect outreach campaign and all future prospect discovery runs. It is the single source of truth for lead state across all agents and tools. No agent writes to Google Sheets directly — all writes go through Python tools in `tools/`.

**Workbook name:** `ZNZB2B — Dozen Lead Database`

**Tabs:**
1. `All Prospects` — 521 source prospects + 7 campaign state columns
2. `Discovery Runs` — new prospects appended by the prospect discovery pipeline
3. `Campaign Stats` — aggregate metrics for reporting and monitoring
4. `Audit Log` — append-only change history

### Enum: contact_status

| Value | Meaning | Who sets it |
|-------|---------|-------------|
| NOT_CONTACTED | No outreach has been sent | Default on migration |
| EMAIL_SENT | At least one email sent; no reply received | send_email.py after confirmed send |
| REPLIED | Prospect replied to any email | inbound_handler.py on reply receipt |
| QUESTION | Prospect asked a product or general question (reply classified as question) | inbound_handler.py after classification |
| QUOTE_REQUEST | Prospect explicitly requested a quote or pricing | inbound_handler.py after classification |
| QUOTE_SENT | Quote has been approved and sent to prospect | quote_sender.py after owner/SM approval |
| UNSUBSCRIBED | Prospect asked to be removed or stop receiving emails | inbound_handler.py on unsubscribe signal |
| BOUNCED_SOFT | Email bounced with a temporary error (e.g. mailbox full) | bounce_handler.py |
| BOUNCED_HARD | Email bounced with a permanent error (e.g. address invalid) | bounce_handler.py |
| NOT_INTERESTED | Prospect replied indicating no interest | inbound_handler.py after classification |

**Status transition rules:**
- NOT_CONTACTED → EMAIL_SENT: only send_email.py may make this transition.
- EMAIL_SENT → REPLIED: any inbound reply triggers this, even if ambiguous.
- REPLIED → QUESTION or QUOTE_REQUEST: classification determines which.
- Any status → UNSUBSCRIBED: unsubscribe signal overrides all other statuses.
- Any status → BOUNCED_HARD: hard bounce overrides. Row is excluded from all future sends.
- BOUNCED_SOFT: row is paused for 7 days, then re-eligible.
- Transitions are append-only in the Audit Log — old status is always recorded.

### Enum: channel

| Value | Meaning |
|-------|---------|
| EMAIL | Contact made via email |
| WHATSAPP | Contact made via WhatsApp Business API (Phase 2) |
| WEB_CHAT | Contact made via website chat widget (Phase 2) |

Phase 1 only populates EMAIL. WHATSAPP and WEB_CHAT are reserved for Phase 2. The column is always present from migration.

### Row-level validation rules (All Prospects tab)

| Rule | Description | On failure |
|------|-------------|-----------|
| no_dup_no | No. must be unique across all rows in the tab | Reject the write; log to Audit Log |
| email_required_for_send | Rows with null Email must have contact_status = NOT_CONTACTED and must never enter the send queue | send_email.py pre-flight check enforces this |
| valid_contact_status | contact_status must be one of the 10 defined enum values | Reject the write; log to Audit Log |
| valid_channel | channel must be one of EMAIL, WHATSAPP, WEB_CHAT, or blank | Reject the write; log to Audit Log |
| valid_boolean | response_flag and quote_requested must be exactly "TRUE" or "FALSE" | Reject the write; log to Audit Log |
| date_format | last_contacted must match YYYY-MM-DD if not blank | Reject the write; log to Audit Log |
| no_manual_state_edit | State columns O–U must only be written by Python tools, never edited manually | Enforced by documentation/process; Audit Log provides detection |

Append-only change history. Every state column write made by any tool must produce a corresponding Audit Log entry. The Audit Log is the authoritative record for debugging, compliance, and PDPA audit purposes.

| A | log_id | Text | Yes | UUID v4 generated at write time. Unique per entry. |
| B | timestamp | Text | Yes | ISO 8601 datetime with UTC timezone: YYYY-MM-DDTHH:MM:SSZ |
| C | prospect_no | Number | Yes | No. value from All Prospects or Discovery Runs. |
| D | tab_name | Text | Yes | "All Prospects" or "Discovery Runs". Which tab was modified. |

| G | new_value | Text | Yes | Value after the write. |
| H | triggered_by_tool | Text | Yes | Name of the Python tool that triggered the write (e.g. "send_email.py", "inbound_handler.py"). |

## Sync Rules (for sync_lead_state.py)

sync_lead_state.py is the only tool that reads AND writes the All Prospects and Discovery Runs tabs in a single operation. All other tools use append-only or targeted cell updates.

### Concurrency

Google Sheets does not provide row-level locks. The following rules prevent race conditions:

1. **Single-writer constraint:** Only one instance of sync_lead_state.py may run at a time. The tool must check for a `_sync_lock` named range (value = "LOCKED" or "UNLOCKED") at the start of execution and set it to "LOCKED" before writing. On completion (success or failure), the tool sets it back to "UNLOCKED". If the lock is already "LOCKED" on startup, the tool exits with a clear error — it does not wait or retry.

2. **Read-modify-write pattern:** The tool reads the entire tab into memory, applies its changes, then writes only the modified rows back using batchUpdate. It does not write the entire sheet on every sync.

3. **State column writes only:** sync_lead_state.py never modifies columns A–N (original prospect data). It only writes columns O–U (state columns).

4. **Stale read protection:** The tool records the spreadsheet's `modifiedTime` before reading. If `modifiedTime` changes between the read and the write, the tool aborts the write, releases the lock, and logs an error. The calling process may retry after a short delay.

### Idempotency

Every write must be idempotent: applying the same operation twice must produce the same result as applying it once.

- For `contact_status` transitions: the tool compares the current value before writing. If the current value equals the intended new value, it skips the write and does not produce an Audit Log entry.
- For `response_flag` and `quote_requested`: same comparison — skip if already the target value.
- For `notes`: append-only. A duplicate note entry is prevented by checking if the exact timestamped string already exists in the cell value.
- For `last_contacted`: write the new date only if it is more recent than the current value.

### Audit Logging

Every write made by sync_lead_state.py (and every other tool) must append an entry to the Audit Log tab immediately after the write succeeds. If the Audit Log write fails, the tool logs the failure to its own run log but does not roll back the primary write. Audit Log failures are surfaced as alerts, not as transaction failures.

### Soft-delete rule

Rows in All Prospects and Discovery Runs are never deleted. Prospects that should be excluded from outreach are marked via contact_status (BOUNCED_HARD, UNSUBSCRIBED) and filtered out by the send queue tools. The record persists forever.
